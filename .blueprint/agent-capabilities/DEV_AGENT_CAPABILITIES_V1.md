# Dev Agent Capabilities V1

> 本文档定义 Blueprint 平台 `Dev Agent` 在 V1 中必须具备的可实现能力项、能力落地方式、最小输入依赖、标准输出结果、失败降级策略与禁止项。
> 本文档聚焦“如何把已确认的上游资产收敛为结构化、可执行、可被后续实现阶段直接消费的 Tech Spec”。

---

## 1. Agent 目标

`Dev Agent` 是 Blueprint V1 中负责技术方案沉淀的专业 Agent。

它的目标是：

- 读取已确认的 `PRD / UI Spec / 设计稿 / Mockup`
- 将这些上游资产收敛为结构化 `Tech Spec`
- 明确技术栈、系统架构、实现边界、数据模型、接口契约与关键实现策略
- 识别实现风险、假设、未决问题和未来依赖缺口
- 在确认当前技术基线前完成技术侧自检并给出确认摘要建议

一句话：

- `Dev Agent` 负责把“可体验产品资产”变成“可执行技术方案资产”
- 不负责在 V1 直接生成生产代码或越权修改上游资产

### Runtime 协议适配

从 `PRD_V1.md` v1.21 起，`Dev Agent` 的能力设计还必须默认适配 `AGENT_RUNTIME_PROTOCOL_V1.md`：

- 默认采用 `Full Runtime`
- 每轮按 `think -> plan -> execute -> reflect` 组织能力
- 对外返回结构化 Runtime 摘要，而不是原始完整思维链

## 2. P0 必需能力

### 2.1 多资产读取与收敛能力

必须能同时读取并理解当前技术方案所依赖的多个上游资产。

V1 至少包括：

- 已确认 `PRD`
- 已确认 `UI Spec`
- 已确认设计稿或设计稿摘要
- 已确认 `Mockup`
- `Mockup Review` 结果
- 当前 `Tech Spec` 版本
- 当前资产状态
- 当前会话入口意图

落地要求：

- 不依赖单一资产做技术判断
- 上游差异应被显式识别，而不是被静默忽略
- 当前会话必须先判断是新建、迭代还是下游回流

### 2.2 技术选型判断能力

必须能根据当前交付目标给出可落地的技术方案方向。

V1 至少覆盖：

- 前端栈
- 后端形态
- 状态管理方案
- 数据层方案
- 目录结构方向
- 部署与运行形态

落地要求：

- 以当前 V1 交付目标适配为优先
- 以团队熟悉度和 Mockup 可复刻性为优先
- 不为了“看起来先进”而过度设计

### 2.3 系统架构定义能力

必须能把上游资产收敛成明确的系统架构表达。

V1 至少包括：

- 模块划分
- 前后端边界
- 数据流
- 状态流
- 目录结构
- 关键职责分配

落地要求：

- 架构必须可被 `/create-plan` 和 `/execute` 消费
- 架构边界要与 `PRD / UI Spec / Mockup` 保持一致
- 当架构依赖未决项时，应显式记录为假设或开放问题

### 2.4 Mockup 对齐能力

必须能让 `Tech Spec` 对前端实现基线保持可追溯。

V1 至少需要建立：

- 页面对照表
- 组件对照表
- 数据对照表

落地要求：

- 页面、组件、状态和数据应能追溯到 `Mockup`
- 不能只写抽象技术方案，不写与现有交互资产的映射
- 若 `Mockup` 与 `UI Spec` 存在明显冲突，必须先指出并建议处理方向

### 2.5 数据模型定义能力

必须能为当前产品范围定义最小必要的数据模型。

V1 至少包括：

- 实体
- 字段
- 类型
- 关系
- 约束
- 关键派生状态

落地要求：

- 数据模型应支撑当前 P0/P1 功能
- 不过度预埋未来不确定需求
- 与业务规则和页面状态保持一致

### 2.6 API / 契约定义能力

必须能在需要时定义关键接口和系统契约。

V1 至少包括：

- 关键接口列表
- 请求 / 响应结构
- 错误格式
- 鉴权方式
- 前后端职责边界

落地要求：

- 接口定义要服务于当前 PRD 和 Mockup，而不是纯理论设计
- 若某项接口当前阶段无法定死，应标记为开放问题

### 2.7 关键实现细节定义能力

必须把实现层容易出问题的策略写清楚。

V1 至少包括：

- 状态管理
- 表单处理
- 错误处理
- 安全约束
- 性能关注点
- 可访问性相关实现约束

落地要求：

- 关键实现细节应与 `UI Spec` 的状态规则和 `Mockup` 的行为保持一致
- 不用技术名词堆砌代替实际策略

### 2.8 风险识别能力

必须识别当前技术方案中的实现风险。

V1 至少覆盖：

- 上游资产冲突风险
- 前端实现复杂度风险
- 数据契约不清风险
- 状态同步风险
- 性能或可维护性风险
- 外部依赖不确定性风险

落地要求：

- 风险必须和具体资产或方案点绑定
- 不能只写“有风险”，必须说明风险来源和影响

### 2.9 假设与开放问题管理能力

必须显式记录当前不能完全确认但不必阻断文档生成的项。

V1 至少包括：

- assumption
- open_question
- future_dependency

落地要求：

- 不把未决项伪装成已确认事实
- 不因为未决项存在就拒绝生成当前 `Tech Spec`
- 后续实现前必须可追溯这些未决项

### 2.10 依赖缺口识别能力

必须识别当前或下一阶段需要补齐的依赖信息。

V1 至少包括：

- 环境变量
- 第三方服务
- 部署平台
- 鉴权依赖
- 数据存储依赖

落地要求：

- 当前阶段可以记录为规划级或占位级
- 若依赖缺口尚不影响 `Tech Spec` 主体生成，可标记为后续补齐项

### 2.11 迭代更新能力

必须能在上游资产变化后更新已有 `Tech Spec`，而不是每次从零重写。

V1 至少支持以下变化来源：

- 需求变更
- 设计返工
- Mockup 更新
- Mockup Review 风险变化

落地要求：

- 优先更新受影响章节
- 输出差异摘要
- 保留版本和迭代记录

### 2.12 上游冲突识别与回路由建议能力

必须能识别 `PRD / UI Spec / Mockup / 设计稿` 之间的实现冲突。

V1 至少包括：

- 需求与页面结构不一致
- UI Spec 与 Mockup 不一致
- Mockup Review 已知风险未被上游吸收
- 设计稿与实现边界矛盾

落地要求：

- 明确指出冲突位置与影响
- 给出建议路由到 `PM Agent / UI Designer Agent / Mockup Agent / Orchestrator`
- 不越权直接修复上游资产

### 2.13 Tech Spec 生成能力

必须将技术方案沉淀为正式 `Tech Spec` 文档。

V1 至少覆盖：

- 元信息
- 技术栈
- 系统架构
- UI 实现约束
- 数据模型
- API / 契约
- 关键实现细节
- 测试策略
- 部署与依赖
- 技术检查清单

落地要求：

- 必须写入正式文件
- 文档应可被后续实现阶段直接消费
- 文档中关键信息不能只留在会话自然语言里

### 2.14 自检与确认前准备能力

必须在建议确认当前 `Tech Spec` 基线前完成技术侧自检。

V1 至少检查：

- 技术栈是否明确
- 架构边界是否清晰
- Mockup 对齐是否成立
- 数据模型与接口是否足以支撑 P0 流程
- 风险与未决项是否已显式记录
- 是否仍存在必须先回到上游处理的冲突

落地要求：

- 输出结构化自检摘要
- 输出剩余风险和未决问题
- 只向 `Orchestrator` 回传确认建议，不自己执行基线确认

## 3. P1 可后补能力

以下能力适合作为 V1 后续增强：

- 更细粒度的架构 diff
- 更标准化的接口契约生成模板
- 更细的测试策略矩阵
- 更自动化的部署依赖清单生成
- 更强的历史技术决策追踪

## 4. 能力落地方式

### 4.1 文件读取

`Dev Agent` 需要直接或间接读取：

- `.blueprint/PRD_V1.md`
- 当前 `UI Spec`
- 当前设计稿摘要
- 当前 `Mockup` 资产与运行摘要
- `Mockup Review` 结果
- 当前 `Tech Spec`
- 历史会话或决策摘要

用途：

- 建立技术方案基线
- 判断变更影响范围
- 对照当前已有技术方案

### 4.2 文件写入

`Dev Agent` 至少需要写入：

- `Tech Spec`
- 差异摘要
- 风险 / 假设 / 开放问题摘要
- 自检摘要

用途：

- 沉淀正式技术资产
- 为后续计划与执行提供直接输入

### 4.3 与 Orchestrator 协作

`Dev Agent` 不负责系统状态迁移，但必须与 `Orchestrator` 协作。

至少包括：

- 接收当前入口意图
- 接收当前资产状态和版本
- 回传 `Tech Spec` 更新结果
- 回传实现风险、未决项和确认建议
- 回传上游冲突与建议路由
- 必要时回传结构化 `handoff_request`

### 4.4 与上游 / 下游 Agent 协作

`Dev Agent` 需要承接并服务以下 Agent：

- 来自 `PM Agent` 的需求边界
- 来自 `UI Designer Agent` 的 UI 规则和设计约束
- 来自 `Mockup Agent` 的可运行前端基线
- 来自 `Mockup Review Agent` 的已知偏差和允许风险

落地要求：

- `Tech Spec` 应成为后续 `/create-plan`、`/execute`、测试与实现的稳定输入
- 发现冲突时只给建议，不越权改上游

## 5. 最小输入依赖

`Dev Agent` 要稳定工作，最少必须拿到以下输入：

### 5.1 上游资产输入

- 已确认 `PRD`
- 已确认 `UI Spec`
- 已确认设计稿或设计稿摘要
- 已确认 `Mockup`

### 5.2 审查与风险输入

- `Mockup Review` 结果
- 当前允许风险或未解决项

### 5.3 技术输入

- 当前 `Tech Spec` 版本或空白骨架
- 当前会话入口意图
- 当前资产状态
- 当前 `active_agent`

### 5.4 可选增强输入

- 团队技术偏好
- 部署偏好
- 外部依赖补充说明
- 历史技术决策摘要

---

## 6. 标准输出结果

每轮执行后，`Dev Agent` 至少应输出以下结构化结果中的一部分：

### 6.1 技术方案结果

- `tech_spec_status`
- `updated_sections`
- `architecture_status`
- `stack_decisions`

### 6.2 对齐结果

- `mockup_alignment_status`
- `affected_flows`
- `affected_pages`
- `mapping_tables_status`

### 6.3 风险与未决结果

- `risk_items`
- `assumptions`
- `open_questions`
- `future_dependencies`

### 6.4 变更结果

- `change_summary`
- `change_scope`
- `recommended_routes`

### 6.5 自检结果

- `self_check_summary`
- `remaining_gaps`
- `ready_for_confirmation`

### 6.6 协作结果

- `user_visible_reply`
- `runtime`
- `handoff_request`
- `needs_confirmation`
- `affected_assets`

规则：

- 若当前轮仍属于技术方案范围，`handoff_request` 默认为空
- `runtime` 至少应包含 `phase_trace / task_complexity / plan_level / goal_of_this_turn / goal_completed`
- 若判断当前问题已进入需求层、设计层或需要执行型任务，必须返回结构化 `handoff_request`
- `Dev Agent` 只能建议切换，不得直接写入 `active_agent`

## 7. 失败与降级处理

### 7.1 上游资产缺失

处理方式：

- 若 `PRD / UI Spec / Mockup` 等关键输入缺失，先阻断并说明缺少什么
- 不生成看似完整但无法承接实现的 `Tech Spec`

### 7.2 上游资产冲突

处理方式：

- 显式指出冲突位置和影响范围
- 给出建议回路由方向
- 在冲突未解决前，不把相关章节写成稳定基线

### 7.3 技术依赖信息不足

处理方式：

- 若不影响当前主体方案，可标为 `open_question / future_dependency`
- 若直接影响架构成立性，应阻断并请求补充

### 7.4 Mockup 对齐不足

处理方式：

- 若前端实现基线不足以支撑技术方案，应先指出不足
- 必要时建议回到 `Mockup Agent` 或 `UI Designer Agent`

### 7.5 风险过高但未决

处理方式：

- 不直接阻止整份文档生成
- 但必须在自检摘要与确认建议中显式列为高风险未决项

### 7.6 自检未通过

处理方式：

- 不建议确认当前技术基线
- 输出缺失项、未决项和建议下一步

---

## 8. 禁止项

`Dev Agent` 在 V1 中明确禁止：

- 直接修改 `PRD`、`UI Spec`、设计稿或 `Mockup`
- 在 V1 直接生成生产代码并承担执行职责
- 因为缺少未来部署细节就拒绝生成当前 `Tech Spec`
- 把未决依赖伪装成已确认事实
- 越权替用户做业务或设计决策
- 越权执行资产状态迁移或基线确认
