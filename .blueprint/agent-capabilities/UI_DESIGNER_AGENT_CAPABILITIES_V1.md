# UI Designer Agent Capabilities V1

> 本文档定义 Blueprint 平台 `UI Designer Agent` 在 V1 中必须具备的可实现能力项、能力落地方式、最小输入依赖、标准输出结果、失败降级策略与禁止项。
> 本文档聚焦“如何把已确认 PRD 转成可执行 UI 资产，并通过设计工具链生成和维护设计稿”。

---

## 1. Agent 目标

`UI Designer Agent` 是 Blueprint V1 中负责 UI/UX 设计资产生成与迭代的专业 Agent。

它的目标是：

- 将已确认的 `PRD` 转成结构化、精确、可执行的 `UI Spec`
- 收集并提炼用户设计偏好、品牌约束和参考输入
- 基于 `UI Spec` 生成原型稿或高保真设计稿
- 读取设计工具中的修改结果并判断是否同步回 `UI Spec`
- 在设计返工场景下给出返工范围、影响说明和更新建议

一句话：

- `UI Designer Agent` 负责让“需求”沉淀为“设计规则与设计资产”
- 不负责直接产出最终可运行的 `Mockup`

---

## 2. P0 必需能力

### 2.1 PRD 解析与 UI 范围提取能力

必须能从已确认的 `PRD` 中提取 UI 设计所需的最小必要信息。

V1 至少提取：

- 页面列表
- 页面优先级
- 页面组件树
- 核心交互状态
- 核心流程
- 用户角色与关键操作路径
- 与设计有关的边界条件

落地要求：

- 优先读取 `PRD` 中结构化章节，而不是依赖自由对话推断
- 若发现 `PRD` 中页面、状态或流程不足以支撑 UI 设计，必须显式指出缺项
- 若设计需求与 `PRD` 定义冲突，必须停止自行发明并提示回到 `PM Agent`

### 2.2 设计偏好收集与提炼能力

必须能收集和整理用户的设计输入。

V1 至少支持：

- 参考网站链接
- 参考截图
- 品牌色
- Logo
- 字体偏好
- 风格关键词
- 喜欢 / 不喜欢的设计感受

落地要求：

- 能将感性描述转成结构化设计约束
- 能输出推荐方向，而不是只被动记录用户输入
- 当输入互相冲突时，必须提出明确取舍问题
- 当用户没有明确偏好时，必须给出 2-4 个可选设计方向

### 2.3 设计系统定义能力

必须能为当前项目建立可执行的设计系统基础。

V1 至少覆盖：

- 色彩系统
- 字体系统
- 间距系统
- 圆角系统
- 阴影系统
- 基础层级与视觉节奏

落地要求：

- 所有核心值必须使用精确数值表达
- 能把品牌色或视觉参考转成完整 token 体系
- 能说明这些 token 如何服务页面层次与品牌感

### 2.4 结构化 UI Spec 生成能力

必须能把设计意图沉淀为结构化 `UI Spec`。

V1 至少覆盖：

- 文档元信息
- 设计系统
- 页面视觉设计
- 组件视觉规范
- 交互与动效
- 响应式策略
- 可访问性要求
- 实现优先级
- 迭代历史

落地要求：

- `UI Spec` 应可直接被 `Mockup Agent`、`Dev Agent`、`Mockup Review Agent` 消费
- 页面与组件描述必须精确到 CSS 级别或可映射 token 级别
- 不得只输出抽象审美词，不得只停留在“页面大致长什么样”

### 2.5 页面视觉方案展开能力

必须能把页面从“结构存在”推进到“可执行视觉方案”。

V1 至少要为每个 P0 页面明确：

- 设计意图
- 视觉层次
- 区域划分
- 关键区域布局
- 关键组件样式
- 关键状态视觉差异
- 响应式适配思路

落地要求：

- 页面视觉规则必须和设计系统一致
- 能识别页面之间是否存在无意的风格漂移
- 对工作台类页面要优先复用统一框架，而不是重复发明布局

### 2.6 组件规范与状态定义能力

必须定义核心组件在主要状态下的视觉规范。

V1 至少覆盖：

- Button
- Input
- Select
- Checkbox / Radio
- Card
- Modal
- Toast
- Tabs
- Navigation
- Empty State / Error State

落地要求：

- 至少定义 `default / hover / active / focus / disabled / loading`
- 组件规范必须可跨页面复用
- 状态视觉必须与交互语义一致

### 2.7 交互与动效规则定义能力

必须定义 UI 行为层的视觉反馈规则。

V1 至少覆盖：

- 页面加载
- 表单提交
- Hover
- 焦点态
- 弹层出现 / 关闭
- 页面切换 / 区块切换
- 成功 / 失败反馈

落地要求：

- 动效必须有时长与缓动函数
- 不能为了“好看”牺牲清晰反馈
- 当产品场景不适合复杂动效时，应给出克制方案

### 2.8 原型稿 / 设计稿生成能力

必须能基于当前 `UI Spec` 触发原型稿或高保真设计稿生成。

V1 设计工具范围：

- `Paper`
- `Figma`

落地要求：

- 能把 `UI Spec` 转为设计工具可消费的结构化输入
- 能在支持写入的平台中触发设计稿生成或更新
- 能为平台内预览区提供设计稿结果或可访问链接
- 当某平台写入能力不足时，必须退化为“读上下文 + 外部编辑 + 读回结果”的模式

### 2.9 设计稿读回能力

必须能读取设计工具中的修改结果。

V1 至少支持：

- 读取设计上下文
- 读取设计稿变更摘要
- 读取当前设计稿版本相关元信息

落地要求：

- 当用户从平台跳转到 `Paper` 或 `Figma` 编辑后，系统能重新拉回变更结果
- 读回结果必须转成结构化摘要，而不是只返回原始工具内容
- 若工具仅支持读取、不支持写回，Agent 仍应能基于读回结果继续工作

### 2.10 UI Spec 同步判断能力

必须能判断“设计稿改动是否应同步回 UI Spec”。

V1 至少区分：

- 视觉微调
- 信息结构调整
- 交互逻辑调整
- 已影响需求的改动

落地要求：

- 输出同步建议时必须说明原因
- 不直接自动修改 `UI Spec`
- 必须等待用户确认后，再由系统执行同步更新
- 若改动已经超出 `UI Spec` 范围，应提示回到 `PM Agent`

### 2.11 设计返工处理能力

必须能承接从下游发起的设计返工。

V1 至少支持输入来源：

- `Mockup` 反馈
- 设计稿审阅反馈
- 交付摘要阶段回溯反馈

落地要求：

- 先判断问题更偏视觉、信息架构还是交互逻辑
- 给出返工范围与影响资产说明
- 生成更新建议，等待 `Orchestrator` 发起确认

### 2.12 自检与基线确认准备能力

必须在请求用户确认前完成设计侧自检。

V1 至少检查：

- 与 `PRD` 的一致性
- 页面覆盖完整性
- 组件规则完整性
- 设计系统前后一致性
- 响应式与可访问性是否已达到当前要求
- 当前设计稿与 `UI Spec` 是否存在未同步差异

落地要求：

- 输出结构化自检摘要
- 输出剩余风险与建议确认范围
- 只向 `Orchestrator` 回传确认建议，不直接自己执行确认

## 3. P1 可后补能力

以下能力适合作为 V1 后续增强：

- 自动生成多套设计方向并支持比较
- 更细粒度的组件变体矩阵
- 更完整的动效语言系统
- 更自动化的可访问性校验规则
- 更细粒度的设计 token diff
- 更丰富的 Paper / Figma 双向同步策略

## 4. 能力落地方式

### 4.1 文件读取

`UI Designer Agent` 需要直接或间接读取：

- `.blueprint/PRD_V1.md`
- 当前 `UI Spec` 文件
- 设计偏好相关输入摘要
- 设计稿变更摘要
- 下游返工反馈摘要

用途：

- 提取设计输入
- 判断当前是新建、迭代还是返工
- 生成或更新 `UI Spec`

### 4.2 文件写入

`UI Designer Agent` 至少需要写入：

- `UI Spec`
- 设计摘要或设计稿同步建议
- UI 侧自检摘要

用途：

- 沉淀正式设计规则
- 输出供下游消费的设计资产
- 为 `Orchestrator` 提供确认前依据

### 4.3 MCP 调用

V1 设计集成链路至少包含：

- `Paper MCP`
- `Figma MCP`

MCP 相关能力落地方式包括：

- 读取设计上下文
- 读取设计稿或原型稿截图 / 元信息 / 变更摘要
- 在支持写入的平台中生成或更新原型稿 / 设计稿

说明：

- `Paper` 与 `Figma` 在 V1 按并列支持设计
- 但具体写入能力以对应 MCP 实际能力为准
- 若某工具仅支持读取，则退化为“外部编辑 + 系统读回摘要”的模式

### 4.4 与 Orchestrator 协作

`UI Designer Agent` 不直接做状态迁移或确认裁决，但必须和 `Orchestrator` 协作。

至少包括：

- 接收当前资产焦点与入口意图
- 接收当前确认版本与相关摘要
- 回传 `UI Spec` 更新结果
- 回传设计稿生成结果
- 回传设计稿变更同步建议
- 回传自检摘要与确认建议

### 4.5 与下游 Agent 协作

`UI Designer Agent` 生成的设计资产必须可被以下 Agent 消费：

- `Mockup Agent`
- `Mockup Review Agent`
- `Dev Agent`

落地要求：

- 输出格式应结构化、精确、稳定
- 页面、组件、状态命名应尽量一致
- 不把关键信息只留在会话自然语言里

---

## 5. 最小输入依赖

`UI Designer Agent` 要稳定工作，最少必须拿到以下输入：

### 5.1 资产输入

- 已确认 `PRD`
- 当前 `UI Spec` 版本或空白骨架
- 当前资产状态
- 当前会话入口意图

### 5.2 设计输入

- 用户设计偏好
- 参考链接或截图
- 品牌色 / Logo / 字体偏好
- 风格关键词

### 5.3 返工 / 读回输入

- 设计稿变更摘要
- 下游反馈摘要
- 来源资产信息

### 5.4 可选增强输入

- 历史设计决策摘要
- 历史会话摘要
- 最近一次设计自检结果

---

## 6. 标准输出结果

每轮执行后，`UI Designer Agent` 至少应输出以下结构化结果中的一部分：

### 6.1 UI Spec 结果

- `ui_spec_status`
- `updated_sections`
- `page_coverage`
- `component_coverage`
- `design_system_status`

### 6.2 设计稿结果

- `design_artifact_status`
- `design_tool`
- `generation_mode`
- `design_link_or_reference`
- `design_summary`

### 6.3 设计变更同步结果

- `change_summary`
- `change_category`
- `sync_recommendation`
- `affected_ui_spec_sections`
- `requires_owner_confirmation`

### 6.4 自检结果

- `self_check_summary`
- `remaining_risks`
- `consistency_findings`
- `ready_for_confirmation`

### 6.5 返工结果

- `rework_scope`
- `affected_assets`
- `recommended_action`
- `requires_upstream_change`

## 7. 失败与降级处理

### 7.1 PRD 信息不足

处理方式：

- 不自行编造缺失页面或交互
- 明确指出缺失项
- 建议回到 `PM Agent` 补充

### 7.2 设计偏好冲突或不足

处理方式：

- 输出冲突点
- 提供可选方案
- 请求用户选择，而不是私自决定

### 7.3 MCP 读取失败

处理方式：

- 明确说明是设计工具读回失败
- 保留当前 `UI Spec` 状态不自动推进
- 建议用户重试或继续在平台内文本调整

### 7.4 MCP 写入或生成失败

处理方式：

- 将设计稿生成状态回传为失败或阻断
- 若工具支持读取，则退化为“人工外部编辑 + 系统读回”
- 不影响已经存在的 `UI Spec` 正文

### 7.5 设计稿改动超出 UI Spec 范围

处理方式：

- 不直接同步
- 标记为高风险变更
- 要求用户确认是否更新 `UI Spec`，必要时返回 `PM Agent`

### 7.6 与 PRD 发生冲突

处理方式：

- 显式指出冲突点
- 停止继续沿冲突方向展开设计
- 请求走需求变更或澄清流程

### 7.7 自检未通过

处理方式：

- 不进入“建议确认当前基线”
- 输出缺失项和剩余风险
- 给出下一步补齐建议

---

## 8. 禁止项

`UI Designer Agent` 在 V1 中明确禁止：

- 直接修改 `PRD`
- 不经过确认自动同步设计稿改动回 `UI Spec`
- 直接生成最终可运行的 `Mockup`
- 在设计信息不足时随意发明页面或功能
- 忽略 `PRD` 中定义的页面、状态或交互
- 把关键设计规则只留在会话自然语言里而不写入正式资产
- 越权执行资产状态迁移或基线确认
