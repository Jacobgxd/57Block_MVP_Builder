# PM Agent Capabilities V1

> 本文档定义 Blueprint 平台 `PM Agent` 在 V1 中必须具备的可实现能力项、能力落地方式、最小输入依赖、标准输出结果、失败降级策略与禁止项。
> 本文档聚焦“如何把模糊产品想法和下游反馈沉淀为结构化、可执行、可确认的 PRD 资产”。

---

## 1. Agent 目标

`PM Agent` 是 Blueprint V1 中负责需求澄清与 `PRD` 资产生成的专业 Agent。

它的目标是：

- 将用户的模糊想法逐步澄清为可执行需求
- 将澄清结果沉淀为结构化 `PRD`
- 在迭代和需求变更场景下更新 `PRD`
- 识别需求缺项、冲突、边界与风险
- 在确认当前需求基线前完成自检并给出确认摘要建议

一句话：

- `PM Agent` 负责把“想法和反馈”变成“稳定需求资产”
- 不负责直接修改设计、Mockup 或技术方案资产

---

## 2. P0 必需能力

### 2.1 项目背景读取能力

必须能读取并理解当前项目的基础背景。

V1 至少包括：

- 项目名称
- 项目描述
- 产品类型 / 行业
- 目标用户
- 目标平台
- 当前 PRD 版本
- 当前资产状态
- 当前会话入口意图

落地要求：

- 开场前先识别当前是新建、迭代还是需求变更
- 不依赖用户重复提供已存在的项目基础信息

### 2.2 需求澄清能力

必须能将模糊描述追问成可执行需求。

V1 至少覆盖：

- 核心问题
- 目标用户
- 成功指标
- 功能目标
- 范围边界
- 优先级
- 验收口径

落地要求：

- 对模糊表达必须继续追问
- 问题默认优先给选项和推荐答案
- 不得把不明确内容伪装成已确认事实

### 2.3 结构化需求拆解能力

必须把澄清后的内容拆成下游可消费的结构。

V1 至少包括：

- 目标与成功指标
- 功能清单与优先级
- 用户故事
- 页面范围
- 核心流程
- 验收标准
- 异常处理
- 关键事件追踪
- 非功能性需求
- 明确不做

落地要求：

- 结构必须稳定，不随对话自由漂移
- 应优先补齐 P0，再延展 P1 / P2
- 当发现某一章节无法成立时，应回退到上游问题继续澄清

### 2.4 PRD 生成能力

必须能将需求内容沉淀为正式 `PRD` 文档。

V1 至少覆盖：

- 概述
- 当前需求全貌
- 迭代历史
- 目标与成功指标
- 功能清单
- 用户故事
- 核心流程图
- 页面原型
- 验收标准
- 异常处理
- 关键事件追踪
- 明确不做
- 非功能性需求

落地要求：

- 必须写入正式文件，而不是只停留在会话回复中
- `当前需求全貌` 要始终保持最新
- `迭代历史` 要记录修改原因与变更内容

### 2.5 核心流程定义能力

必须能把核心需求转成流程表达。

V1 至少包括：

- 正常路径
- 关键分支判断
- 异常出口
- 涉及角色
- 关联功能

落地要求：

- 使用 `Mermaid` 表达
- 流程图不能只写 happy path
- 若流程本身定义不清，必须先追问再生成

### 2.6 页面原型抽象能力

必须能把需求范围推进到页面层定义。

V1 至少包括每个 P0/P1 页面：

- ASCII 线框图
- 组件树
- 交互与状态表

落地要求：

- 页面原型必须服务后续 `UI Designer Agent`
- 页面结构与 PRD 功能边界必须一致
- 若页面层信息不足，必须追问而不是随意补齐

### 2.7 验收标准定义能力

必须能把需求写成可验证标准。

V1 至少覆盖：

- 正常流程
- 失败流程
- 边界条件

落地要求：

- 使用 `Given-When-Then`
- 验收标准应能被 `Mockup` 与后续 `Test Spec` 消费

### 2.8 异常与边界条件梳理能力

必须能系统性定义异常处理和边界场景。

V1 至少包括：

- 输入异常
- 网络 / 系统异常
- 业务异常
- 边界情况

落地要求：

- 区分功能级异常和全局异常
- 异常处理需要同时说明用户提示与系统行为

### 2.9 事件追踪与非功能需求定义能力

必须补齐需求中容易遗漏的运营与系统约束。

V1 至少包括：

- 埋点事件
- 转化漏斗
- 关键监控指标
- 性能
- 安全
- 可访问性
- 兼容性

落地要求：

- 当用户没给明确值时，可给默认建议，但要可追溯
- 不得直接跳过这些章节

### 2.10 需求变更吸收能力

必须能承接从下游资产发起的需求变更。

V1 至少支持来源：

- `UI Spec`
- `设计稿`
- `Mockup`
- `Tech Spec`
- `交付摘要`

落地要求：

- 先理解“为什么回来改 PRD”
- 区分原始需求遗漏、定义不清、业务方向调整
- 输出更新建议而不是直接跳过澄清

### 2.11 影响分析输入准备能力

必须为跨资产影响分析提供需求侧输入。

V1 至少包括：

- 哪些功能定义被改动
- 哪些页面定义会受影响
- 哪些下游资产可能失效
- 是否需要更新当前基线

落地要求：

- `PM Agent` 可以提出影响分析建议
- 最终影响同步与状态迁移由 `Orchestrator` 执行

### 2.12 自检与确认前准备能力

必须在请求用户确认当前 PRD 基线前完成需求侧自检。

V1 至少检查：

- 目标是否清晰
- 成功指标是否可量化
- 功能范围是否完整
- 核心流程是否存在
- 页面原型是否覆盖关键页面
- 验收标准是否可验证
- 是否存在明显待确认项

落地要求：

- 输出结构化自检摘要
- 输出剩余风险和待确认点
- 只向 `Orchestrator` 回传确认建议，不自己执行状态确认

## 3. P1 可后补能力

以下能力适合作为 V1 后续增强：

- 更细粒度的需求 diff
- 更自动化的冲突检测
- 更标准化的优先级决策辅助
- 跨会话需求主题聚类
- 更细的需求版本对比摘要

## 4. 能力落地方式

### 4.1 文件读取

`PM Agent` 需要直接或间接读取：

- `.blueprint/PRD_V1.md`
- 项目基础信息摘要
- 历史会话摘要
- 下游反馈摘要
- 当前会话入口意图

用途：

- 判断当前会话模式
- 理解已有需求上下文
- 决定本轮修改范围

### 4.2 文件写入

`PM Agent` 至少需要写入：

- `PRD`
- `迭代历史`
- 当前轮需求变更摘要
- 自检摘要

用途：

- 沉淀正式需求资产
- 让下游 Agent 始终读取最新需求全貌

### 4.3 与 Orchestrator 协作

`PM Agent` 不负责系统状态迁移，但必须与 `Orchestrator` 协作。

至少包括：

- 接收当前入口意图
- 接收当前 PRD 状态与版本
- 回传 PRD 更新结果
- 回传缺项与风险摘要
- 回传是否建议确认当前基线
- 回传需求变更对下游资产的影响建议

### 4.4 与下游 Agent 协作

`PM Agent` 产出的 PRD 需要被以下 Agent 消费：

- `UI Designer Agent`
- `Mockup Agent`
- `Mockup Review Agent`
- `Dev Agent`

落地要求：

- `PRD` 内容应结构化、稳定、可追踪
- 不把关键信息只留在会话自然语言中

---

## 5. 最小输入依赖

`PM Agent` 要稳定工作，最少必须拿到以下输入：

### 5.1 项目输入

- 项目基础信息
- 当前 PRD 版本
- 当前资产状态
- 当前会话入口意图

### 5.2 需求输入

- 用户当前轮输入
- 历史需求摘要
- 当前待确认项

### 5.3 变更输入

- 下游反馈摘要
- 来源资产
- 当前受影响资产范围

### 5.4 可选增强输入

- 历史版本摘要
- 最近一次自检结果
- 最近一次影响分析摘要

---

## 6. 标准输出结果

每轮执行后，`PM Agent` 至少应输出以下结构化结果中的一部分：

### 6.1 需求澄清结果

- `clarified_topics`
- `open_questions`
- `assumptions`
- `confidence_level`

### 6.2 PRD 更新结果

- `prd_status`
- `updated_sections`
- `change_summary`
- `iteration_entry`

### 6.3 风险与缺项结果

- `missing_items`
- `risk_items`
- `conflict_items`

### 6.4 变更结果

- `change_type`
- `affected_assets`
- `recommended_followups`

### 6.5 自检结果

- `self_check_summary`
- `remaining_gaps`
- `ready_for_confirmation`

## 7. 失败与降级处理

### 7.1 项目背景不足

处理方式：

- 若缺少项目基础信息，先追问补齐
- 不直接生成看似完整的 PRD

### 7.2 需求表述过于模糊

处理方式：

- 不直接填充为正式需求
- 先输出待确认点并继续追问

### 7.3 需求间冲突

处理方式：

- 显式指出冲突点
- 要求用户做取舍
- 冲突未解前，不应把冲突内容写成稳定基线

### 7.4 页面或流程信息不足

处理方式：

- 不编造页面或流程
- 标记缺口
- 继续补充澄清

### 7.5 需求变更超出当前理解范围

处理方式：

- 先收敛变更范围
- 必要时拆成多轮澄清
- 不一次性把大范围变化直接写入正式 PRD

### 7.6 自检未通过

处理方式：

- 不建议确认当前基线
- 输出缺失项与剩余风险
- 给出下一步补齐方向

---

## 8. 禁止项

`PM Agent` 在 V1 中明确禁止：

- 直接修改 `UI Spec`、设计稿、`Mockup` 或 `Tech Spec`
- 在信息不足时编造需求
- 跳过关键确认直接把内容写成已确认基线
- 删除用户已确认需求，除非用户明确要求
- 把待确认内容混入正式需求且不标记
- 越权执行资产状态迁移
- 直接决定跨资产更新执行结果
