# Mockup Agent Capabilities V1

> 本文档定义 Blueprint 平台 `Mockup Agent` 在 V1 中必须具备的可实现能力项、能力落地方式、最小输入依赖、标准输出结果、失败降级策略与禁止项。
> 本文档聚焦“如何把已确认的上游资产转成可运行、可预览、可重生成的 Web Mockup”。

---

## 1. Agent 目标

`Mockup Agent` 是 Blueprint V1 中最接近执行链路的系统内部 Agent。

它的目标是：

- 基于已确认的 `PRD / UI Spec / 设计稿` 生成可运行的 Web Mockup
- 让用户能够预览、点击、浏览核心流程，而不是只看静态页面
- 在上游资产变化或 Review 问题出现后，执行局部更新或重生成
- 将执行结果、预览地址、风险项和失败原因回传给 `Orchestrator`

一句话：

- `Mockup Agent` 负责把“文档资产”变成“可体验的前端原型”
- 不负责定义需求，也不负责裁决设计方向

### Runtime 协议适配

从 `PRD_V1.md` v1.21 起，`Mockup Agent` 的能力设计还必须默认适配 `AGENT_RUNTIME_PROTOCOL_V1.md`：

- 默认采用执行型 `Full Runtime`
- 每轮按 `think -> plan -> execute -> reflect` 组织能力
- 对外返回结构化 Runtime 摘要，而不是原始完整思维链

## 2. P0 必需能力

### 2.1 上游资产读取能力

必须能读取并理解当前 Mockup 所依赖的最小上游资产。

V1 至少包括：

- 已确认 `PRD`
- 已确认 `UI Spec`
- 已确认设计稿或设计稿摘要
- 当前生成参数
- 当前版本信息

落地要求：

- 优先读取结构化资产，不依赖会话自然语言作为主要输入
- 能识别当前输入版本，并在执行结果中记录本次依赖版本
- 当 `PRD / UI Spec / 设计稿` 之间出现冲突且无法判断主线时，必须回报 `Orchestrator`

### 2.2 Mockup 范围提取能力

必须能从上游资产中提取出需要被实现为 Mockup 的最小可体验范围。

V1 至少提取：

- 页面清单
- 路由结构
- 核心流程
- 页面之间的导航关系
- 关键组件规范
- 关键交互状态
- 关键空状态 / 错误状态

落地要求：

- 优先覆盖 P0 页面与 P0 流程
- 不能把“有文档描述但没做成可体验路径”的内容当作已经交付
- 当页面或流程定义不足以搭起可点击原型时，应视为关键缺口

### 2.3 基线识别能力

必须能识别本次 Mockup 构建基于哪组上游基线。

V1 至少识别：

- `PRD` 版本
- `UI Spec` 版本
- 设计稿版本或摘要版本
- 当前 Mockup 版本
- 本次执行模式

落地要求：

- 执行结果中必须写明 `input_versions`
- 当上游资产处于 `待同步` 或未确认状态时，不得把其当作稳定基线继续推进

### 2.4 生成模式判定能力

必须能判断本次执行属于哪一种生成模式。

V1 至少支持：

- `initial_generate`
- `incremental_update`
- `regenerate`

落地要求：

- 首次生成时创建完整可运行入口
- 局部更新时只更新受影响页面、组件、样式或 mock 数据
- 重生成时按变更摘要、Review 问题或影响范围决定重建范围

### 2.5 Mockup 架构搭建能力

必须能将上游资产转成前端原型工程结构。

V1 至少包括：

- 页面文件结构
- 路由结构
- 全局样式或 design token 映射
- 基础组件
- 页面级组件
- mock 数据层
- 状态切换入口

落地要求：

- 工程结构必须可后续迭代，而不是一次性拼接页面
- 应优先复用 `UI Spec` 中的组件规范和工作台统一框架
- 不允许为了赶生成直接输出难以重生成的混乱结构

### 2.6 视觉落地能力

必须能把 `UI Spec` 和设计稿中的视觉规则落到 Mockup 中。

V1 至少覆盖：

- 设计 token 映射
- 页面布局
- 核心组件样式
- 关键区域层次
- 主要状态的视觉差异

落地要求：

- 视觉还原优先级应服从 `PRD -> UI Spec -> 设计稿` 的主线关系
- 设计稿有明确视觉落点时，不应随意偏离
- 若设计稿缺少局部细节，可在不影响主流程的前提下按 `UI Spec` 规则补足，并记录为 `fallback_items`

### 2.7 交互流程实现能力

必须能让 Mockup 体现核心流程，而不仅仅展示页面。

V1 至少支持：

- 页面切换
- 导航跳转
- 关键 CTA 点击
- 表单提交的前端反馈
- 成功 / 失败状态切换
- 核心流程的主要路径

落地要求：

- 关键流程要可走通
- 交互反馈要能被用户真实感知
- 不接真实后端，但不能因为使用 mock 数据就省略核心交互逻辑

### 2.8 Mock 数据组织能力

必须为 Mockup 提供足够支撑体验的静态数据或假数据。

V1 至少覆盖：

- 默认列表数据
- 详情数据
- 空状态数据
- 错误状态数据
- 表单默认值或样例输入

落地要求：

- mock 数据要服务于核心流程验证
- mock 数据结构要能支撑后续局部更新
- 不得伪装成真实后端数据接入

### 2.9 本地运行能力

必须能让生成结果在本地运行。

V1 至少包括：

- 安装必要依赖
- 启动本地开发预览
- 检查关键入口是否可访问
- 记录预览地址

落地要求：

- 若本地运行成功，必须返回明确 `preview_url`
- 若命令执行失败，必须返回失败摘要而不是静默结束
- 若已存在可复用运行环境，优先复用而不是重复创建

### 2.10 预览交付能力

必须把 Mockup 结果变成可供用户和系统消费的预览资产。

V1 至少包括：

- 平台内预览可用链接
- 统一入口页面
- 可浏览路径说明
- 本次生成范围说明

落地要求：

- 预览结果应支持用户校验核心流程
- `Orchestrator` 应能基于回传结果判断是否进入 `Mockup Review`

### 2.11 基础验证能力

生成后必须先做最小验证，不能把完全不可用结果直接往下传。

V1 至少检查：

- 项目结构是否完整
- 页面入口是否存在
- 路由是否可访问
- 基础组件是否可渲染
- 关键交互是否可触发
- 本地运行是否成功

落地要求：

- 验证结果应写入结构化执行回执
- 验证失败时要区分 `failed / partial_success / blocked`

### 2.12 增量更新能力

必须优先支持局部更新，而不是每次全量重建。

V1 至少支持以下变化类型：

- 页面布局变化
- 组件规范变化
- design token 变化
- 核心流程变化
- 文案微调
- 设计稿局部更新

落地要求：

- 能识别受影响范围
- 只更新目标文件和必要关联文件
- 不静默覆盖无关文件

### 2.13 Review 问题吸收能力

必须能读取 `Mockup Review Agent` 返回的问题并转化为执行范围。

V1 至少支持：

- 页面缺漏
- 流程断点
- 状态缺失
- 视觉漂移
- 组件实现偏差

落地要求：

- 能将问题清单映射到页面 / 组件 / 样式 / 路由层
- 能根据问题严重度决定是局部修复还是重生成
- 不能绕过 `Orchestrator` 直接自行决定高影响范围重建

### 2.14 风险与缺口分级能力

必须区分哪些缺口可以降级继续，哪些必须阻断。

V1 至少区分：

- 关键缺口
- 非关键缺口

关键缺口示例：

- 核心页面清单不足
- 核心流程无法建立
- 设计规范严重冲突

非关键缺口示例：

- 个别占位图缺失
- 次要 hover 动效缺失
- 次要文案未最终确定

落地要求：

- 关键缺口返回 `blocked`
- 非关键缺口允许降级，但必须写入 `fallback_items`

### 2.15 结构化执行结果回传能力

必须把执行结果以结构化形式回传给 `Orchestrator`。

V1 至少包含：

- `status`
- `mode`
- `input_versions`
- `output_path`
- `preview_url`
- `generated_scope`
- `affected_files`
- `fallback_items`
- `risk_items`
- `error_summary`

落地要求：

- 返回结果必须支持后续 `Mockup Review`、状态迁移和再生成
- 不得只返回自然语言描述

## 3. P1 可后补能力

以下能力适合作为 V1 后续增强：

- 更细粒度的页面级 diff 识别
- 更自动化的交互冒烟测试
- 更丰富的 mock 数据场景生成
- 多入口预览说明与场景切换
- 更精细的生成 manifest
- 对历史 Mockup 版本的回看与比较支持

## 4. 能力落地方式

### 4.1 文件读取

`Mockup Agent` 需要直接或间接读取：

- `.blueprint/PRD_V1.md`
- 当前 `UI Spec`
- 当前设计稿摘要或设计稿上下文结果
- 历史 Mockup 工程文件
- Review 问题清单
- `Orchestrator` 注入的变更摘要或生成参数

用途：

- 提取生成范围
- 对比当前版本
- 执行局部更新

### 4.2 文件写入

`Mockup Agent` 至少需要写入：

- `.blueprint/mockup/` 下的前端原型文件
- 运行摘要
- 生成结果记录
- 预览地址记录

用途：

- 形成正式 Mockup 资产
- 支撑后续 Review 与再生成

### 4.3 命令执行

`Mockup Agent` 是当前 V1 中最明确需要命令执行能力的 Agent 之一。

V1 至少涉及：

- 安装或校验依赖
- 启动本地预览服务
- 构建项目
- 检查运行状态

说明：

- 命令执行结果必须被结构化回传
- 若已有健康运行实例，应优先复用，避免重复启动
- 长运行任务需要被 `Orchestrator` 追踪状态

### 4.4 与 Orchestrator 协作

`Mockup Agent` 不直接做系统裁决，但必须与 `Orchestrator` 协作。

至少包括：

- 接收执行模式与生成参数
- 接收当前资产状态和版本信息
- 回传结构化执行结果
- 在阻断时回传明确失败原因
- 在重生成场景中回传受影响范围
- 必要时回传结构化 `handoff_request`

### 4.5 与 Mockup Review Agent 协作

`Mockup Agent` 与 `Mockup Review Agent` 构成生成-检查闭环。

至少包括：

- 接收 Review 问题清单
- 将问题清单映射为受影响范围
- 修复后再次产出可检查结果
- 回传本轮修复范围与残余风险

## 5. 最小输入依赖

`Mockup Agent` 要稳定工作，最少必须拿到以下输入：

### 5.1 上游资产输入

- 已确认 `PRD`
- 已确认 `UI Spec`
- 已确认设计稿或设计稿摘要

### 5.2 执行输入

- 当前执行模式
- 当前版本信息
- 当前生成范围
- 当前变更摘要或 Review 问题清单
- 当前 `active_agent`

### 5.3 运行输入

- Mockup 工程路径
- 本地运行方式或命令约定
- 必要依赖信息

### 5.4 可选增强输入

- 历史 Mockup 版本
- 历史执行结果
- 上一次失败摘要

---

## 6. 标准输出结果

每轮执行后，`Mockup Agent` 至少应输出以下结构化结果中的一部分：

### 6.1 执行状态结果

- `status`
- `mode`
- `input_versions`
- `generated_scope`
- `affected_files`

### 6.2 交付结果

- `output_path`
- `preview_url`
- `entry_points`
- `run_summary`

### 6.3 风险与降级结果

- `fallback_items`
- `risk_items`
- `blocked_reasons`
- `error_summary`

### 6.4 再生成结果

- `regenerated_scope`
- `resolved_review_items`
- `remaining_review_risks`

### 6.5 协作结果

- `runtime`
- `handoff_request`
- `needs_confirmation`
- `affected_assets`

规则：

- 默认情况下，`handoff_request` 为空
- `runtime` 至少应包含 `phase_trace / task_complexity / plan_level / goal_of_this_turn / goal_completed`
- 若执行中发现问题本质上属于需求缺口、设计冲突或需先由系统确认范围，必须返回结构化 `handoff_request`
- `Mockup Agent` 只能建议切换，不得直接写入 `active_agent` 或触发上游资产更新

## 7. 失败与降级处理

### 7.1 上游资产缺失

处理方式：

- 若 `PRD / UI Spec / 设计稿` 任一关键输入缺失，直接返回 `blocked`
- 明确指出缺少哪个输入，不能继续生成

### 7.2 上游资产冲突

处理方式：

- 若冲突影响主流程或视觉主线，直接阻断
- 回传冲突来源，由 `Orchestrator` 决定是否回到上游处理

### 7.3 生成范围无法判断

处理方式：

- 若无法准确定位受影响范围，可升级为更大范围重生成
- 若连重生成范围都无法判断，应返回阻断

### 7.4 本地构建失败

处理方式：

- 返回 `failed`
- 提供错误摘要
- 不得伪装成“已生成但待修复”

### 7.5 本地运行失败

处理方式：

- 返回 `failed`
- 说明是运行失败而非生成失败
- 若构建成功但启动失败，应保留输出路径并单独记录运行问题

### 7.6 局部页面失败

处理方式：

- 若核心流程仍可运行，可返回 `partial_success`
- 必须明确失败页面、组件或状态范围

### 7.7 非关键缺口降级

处理方式：

- 允许使用默认值继续生成
- 但必须记录到 `fallback_items`
- 不得将默认值伪装成已确认设计

### 7.8 Review 问题未完全修复

处理方式：

- 允许返回“已修复部分问题”的结果
- 但必须明确剩余问题，不得自动推进为可确认基线

## 8. 禁止项

`Mockup Agent` 在 V1 中明确禁止：

- 直接修改 `PRD`、`UI Spec` 或设计稿正文
- 在需求归因不清时自己决定回写上游文档
- 把静态截图或纯展示页面冒充为可交互 Mockup
- 跳过本地运行或基础验证直接回传成功
- 在局部变化场景中无理由全量重建整个项目
- 静默覆盖无关文件
- 把默认值或猜测内容伪装成已确认实现
- 直接与用户进行长期人格化共创对话
